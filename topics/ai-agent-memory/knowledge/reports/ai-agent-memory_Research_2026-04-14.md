### 3. 【架构实测】三层记忆系统已成行业标准

**来源**: [SegmentFault 2026-01-09](https://segmentfault.com/a/1190000047532672)、[CSDN 2026-03-12](https://blog.csdn.net/sgr011215/article/details/158935991)

三层记忆架构遵循 **"推理即时性 → 会话连贯性 → 跨会话持久性"** 的逻辑：

```
┌─────────────────────────────────────────────┐
│  L0 · 工作记忆（Working Memory）              │
│  延迟: <1ms  │  容量: 受限于上下文窗口           │
│  存储: 当前任务状态、决策日志                   │
├─────────────────────────────────────────────┤
│  L1 · 向量语义记忆（Vector Memory）            │
│  延迟: <10ms │  容量: 可扩展                    │
│  存储: 语义相似检索、对话摘要                   │
├─────────────────────────────────────────────┤
│  L2 · 持久化长期记忆（Persistent Memory）       │
│  延迟: ~100ms│  容量: 理论无限                  │
│  存储: 用户偏好、跨会话事实、经验知识            │
└─────────────────────────────────────────────┘
```

OpenClaw 框架实测采用 **SQLite + 向量数据库** 的组合实现三层记忆。

> **钻头评**: 三层架构不是"过度设计"，而是不同层次对**速度、容量、精度**的天然权衡需求。刚需场景用 L0，追求个性化用 L2，缺一不可。

### 5. 【技术细节】LLM 为什么"记不住"？Token 经济的博弈

**来源**: [腾讯云 2026-04-14](https://cloud.tencent.com/developer/article/2640506)

LLM 本质是**无状态**的（Stateless）——每次请求独立，节省算力。要让 Agent "记住"，有三条技术路径：

| 方案 | 原理 | 成本 | 效果 |
|------|------|------|------|
| **messages 数组** | 把历史对话塞进 Prompt | 高（按 token 收费） | 会话内有效 |
| **截断（Truncation）** | 只保留最近几轮 | 中 | 丢失历史上下文 |
| **摘要（Summarization）** | LLM 压缩历史为摘要 | 中 | 可能丢失细节 |
| **RAG（检索增强）** | 向量检索补充上下文 | 低 | 精准、动态 |
| **外部记忆系统** | 独立存储层持久化 | 低（存储成本） | 跨会话、个性化 |

> **钻头评**: 记忆的本质是**成本与效果的博弈**。RAG + 外部记忆的组合（Mem0 路线）正成为主流——用 embedding 检索替代全量上下文，是 Token 经济下的最优解。
